  2. tutorial: creating an SDL project
    • file new project
      • Win32 console application
      • do not tick the create solution directory box
      • the path should be the path to your personal folder
      • tip: choose a name without spaces (use underscore to break words)
      • select empty project
    • choose properties on project
      • choose configuration: all configurations
      • choose VC++ directories
      • choose include directories
      • choose edit and add:
        • $(USERPROFILE)\Documents\programming\libraries\include
      • choose library directories
      • choose edit and add:
        • $(USERPROFILE)\Documents\programming\libraries\lib
      • open the Linker options
      • choose input
      • choose additional dependencies
      • choose edit and add:
        • SDL.lib
        • SDLmain.lib
        • SDL_mixer.lib
        • SDL_ttf.lib
        • opengl32.lib
      • tip: to prevent a linker warning in the debug build:
        • choose configuration: debug
        • choose ignore specific default libraries
        • choose edit and add: msvcrt.lib
    • you can now add files to your project
    • to test this:
      • take any existing project
      • copy in all cpp source and header files
      • choose add existing item
      • copy in any required data files (bitmaps, audio, etc.)
      • now build and run
      • if it works you can now build your new project

  4. lecture: construction topics
    • programming by intention
      • wishful thinking
      • divide and conquer
      • what if somebody had already written the code I want?
    • assertions
      • used to catch programming mistakes
      • don't use for error handling
    • pair programming
      • two programmers working together at one machine
      • good for team working - spreading knowledge
      • raises the "truck number" of a team
      • improves code quality
      • lowers development time
    • unit testing
      • writing another program to test the parts of a program
      • helps to keep the code modular
      • good for developing engines and modules
      • makes refactoring safer
      • some external tool support
    • refactoring
      • already seen this in previous examples
      • change the expression without changing the behaviour
      • reduce duplication
      • improve modularity
      • make changes easier
      • improve readability
    • debug build
      • runs slower but checks for more errors
      • easier to debug
      • conditional compilation
    • DRY: don't repeat yourself
      • keep an eye on areas that you are frequently repeating
      • are you doing a lot of cut and paste programming?
      • refactor to remove duplication
    • YAGNI: you aren't gonna need it
      • write your program under this assumption
      • don't add functionality for things you don't need yet!
    • KISS: keep it simple, stupid!
      • simple programs are much easier to write
      • refactor to remove unnecessary functions, parameters, or variables

  8. tutorial: building on a framework
    • start with a new SDL project as previous tutorial
    • remove game specific parts to leave just the framework
    • identify the player object in your own game
      • this is the thing that will respond to your input
      • it might be: a spaceship; a man; or, something more abstract
      • programming by intention: assume you already had this object
      • create a player source file and header file
      • draw your player object
      • make it update and respond to your input
      • identify other objects in your game and repeat this process
